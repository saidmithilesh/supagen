# Model Catalog Owns Canonical Provider And Model Language

Supagen treats Model Catalog as a first-class domain with canonical concepts for AI providers, AI gateways, model authors, upstream serving providers, catalog models, and provider model offerings. The initial catalog will be bootstrapped from OpenRouter data because OpenRouter gives broad model coverage quickly, but OpenRouter DTOs, terminology, and request-shaping details must stay inside the OpenRouter infrastructure adapter and sync/import layer rather than becoming Supagen's core catalog model. This prevents OpenRouter lock-in while still allowing Supagen to use OpenRouter-specific gateway routing metadata when making v1 invocations.

While OpenRouter is the near-term primary catalog source of truth, Supagen may persist fetched OpenRouter API responses or source snapshots for caching, sync provenance, debugging, and re-import workflows. Those source records are not the canonical catalog store: public catalog reads, invocation validation, routing decisions, and future cost workflows should operate from Supagen-shaped catalog records derived from the source data. OpenRouter-shaped fields may be retained as external references or source metadata, but they must not define the main database schema, application service contracts, or frontend API response shape.

For the initial OpenRouter-backed experiment, Supagen only includes OpenRouter catalog models that currently have endpoint data because `endpoint: null` means OpenRouter no longer serves that model. Detailed provider-offering information should come from OpenRouter's per-model endpoint stats API and be transformed into Supagen canonical provider-offering shapes rather than inferred only from the catalog list response.
